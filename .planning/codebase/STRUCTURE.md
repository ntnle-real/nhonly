## Directory Purposes

**C3/kernel/ (CRITICAL FOR UNDERSTANDING):**
- Purpose: Local, self-contained C3 contract kernel with theory, templates, and examples
- Contains: Markdown documentation defining contract patterns, observation API, and contract templates for TypeScript/Svelte/JavaScript/Python/SQL
- Key files: THEORY.md (conceptual foundation), CONTRACTS.md (copy-paste templates), obs_api.md (API reference)
- Examples: `EXAMPLES/typescript/transform.ts` shows contract structure; `EXAMPLES/svelte/FormValidator.svelte` shows Svelte component pattern
- **NEVER reference external C3** — this kernel is complete and self-contained

**src/lib/ (Business Logic Layer):**
- Purpose: Encapsulate application services as reusable, contract-bound functions
- Contains: Three domain modules (recording, archive, i18n) exported as functions
- Key files:
  - `recording.ts` (lines 1-91): Web Audio API wrapper with startRecording, stopRecording, getElapsedMs, isRecording
  - `archive.ts` (lines 1-128): IndexedDB wrapper with initDatabase, saveStory, getAllStories, getStory
  - `i18n.ts` (lines 1-69): Language store and translation getter

**src/routes/ (UI & Page Layer):**
- Purpose: SvelteKit page components handling user interactions and routing
- Contains: Single-page application with all features on main page
- Key files:
  - `+page.svelte` (lines 1-163): Main application UI with state machine (idle → recording → stopped)
  - `+layout.svelte` (lines 1-12): Root wrapper applying favicon and rendering children

**static/ (Public Assets):**
- Purpose: Static files served as-is without processing
- Contains: robots.txt for SEO

**C3/kernel/EXAMPLES/ (Reference Implementation Patterns):**
- Purpose: Show how to write contracts in each language
- TypeScript pattern in `EXAMPLES/typescript/transform.ts` (lines 25-57): Contract shape with serves, declares, succeeds_if, fails_if
- Svelte pattern in `EXAMPLES/svelte/FormValidator.svelte` (lines 1-80): Component with obs call pattern
- **Copy these patterns into feature code** — they're starting points, not libraries

## Naming Conventions

**Files:**
- `.svelte` for Svelte components
- `.ts` for TypeScript modules
- `.md` for documentation (kernel docs are markdown-based)
- Route files follow SvelteKit convention: `+page.svelte`, `+layout.svelte`

**Directories:**
- `src/lib/` for reusable services and utilities
- `src/routes/` for SvelteKit page components
- `C3/kernel/` for C3 theory and templates (never modify)
- `C3/EXAMPLES/` or `C3/kernel/EXAMPLES/` for reference implementations

**Functions:**
- camelCase: `startRecording`, `saveStory`, `initDatabase`, `setLanguage`
- Getters with no side effects: `getElapsedMs`, `getAllStories`, `getStory`
- Boolean predicates with `is`: `isRecording`

**Types/Interfaces:**
- PascalCase: `RecordingSession`, `ArchivedStory`, `Language`
- Type union: `'idle' | 'recording' | 'stopped'` for UI state

**Constants:**
- UPPER_SNAKE_CASE: `DB_NAME`, `DB_VERSION`, `STORE_NAME`

**Variables:**
- camelCase: `recordingState`, `audioBlob`, `mediaRecorder`, `currentLanguage`
- Svelte stores prefixed with `$`: `$currentLanguage`, `$state()`

## Where to Add New Code

**New Feature Implementation:**
- **Primary code:** Add to `src/lib/{feature}.ts` following contract pattern from `C3/kernel/CONTRACTS.md`
- **UI for feature:** Add handlers and state to `src/routes/+page.svelte` or create new route in `src/routes/`
- **Contract pattern:** Must declare MARK header, Purpose, Success, Failure before implementation (see `src/lib/recording.ts` lines 1-3 for example)

**New Route/Page:**
- Create `src/routes/[route]/+page.svelte` following SvelteKit file-based routing
- Reference `C3/kernel/EXAMPLES/svelte/` for component structure with contract markers

**Reusable Component:**
- Create `src/lib/components/{ComponentName}.svelte`
- Export with contract declaration matching template from `C3/kernel/CONTRACTS.md`

**Utility Functions:**
- Add to `src/lib/index.ts` or create new `src/lib/{utility}.ts`
- Follow contract pattern from `C3/kernel/CONTRACTS.md` TypeScript section

**Test Files:**
- Create alongside source: `src/lib/{module}.test.ts`
- Follow testing patterns from TESTING.md (not yet established)

## Special Directories

**C3/kernel/ (Generated/Protected):**
- Purpose: C3 contract-driven development kernel (do NOT modify)
- Generated: No (hand-authored reference implementation)
- Committed: Yes (essential for understanding contract patterns)
- Read-only: Should be treated as reference, not modified for project-specific needs

**.svelte-kit/ (Generated):**
- Purpose: SvelteKit build artifacts and type generation
- Generated: Yes (by `npm run check` and build)
- Committed: No (in .gitignore)
- Auto-regenerated by SvelteKit during development

**.planning/codebase/ (Generated by GSD):**
- Purpose: Machine-readable architecture/conventions/testing documentation
- Generated: Yes (by `/gsd:map-codebase` command)
- Committed: Yes (useful for team understanding)
- Contains: ARCHITECTURE.md, STRUCTURE.md, CONVENTIONS.md, TESTING.md, CONCERNS.md

## Growth Points & Future Structure

- Extract recording controls to `src/lib/components/RecordingControl.svelte`
